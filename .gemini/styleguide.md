# Review style guide

Review every pull request against the conventions below. These are hard project rules, not preferences.

## Focus

Flag what tooling cannot catch: semantic correctness, duplicated logic, broken skeleton symmetry, wrong module placement, missing wiring or paired files, and unclear design. ESLint, Prettier, and CI already enforce the mechanical rules — the `.js` import extension, `import type`, the source-comment ban, and formatting. Treat those as background context, not review comments. Re-flagging a lint-enforced rule is noise.

## Imports and modules

- Import order in a rule file: external `@typescript-eslint/utils` imports first, then local shared-module imports from `core/`, `ast/`, `fixers/`, and `text/`. Flag an order that breaks from sibling rules.
- The package is NodeNext with `verbatimModuleSyntax`, so relative imports end in `.js` and type-only imports use `import type`. ESLint enforces both; do not comment on them.

## Shared layer

- Cross-rule logic lives in five domain folders under `src/`, named after the value each module operates on: `core/` for the `createRule` factory, `constants/` for one named constant per file, `ast/` for TSESTree helpers, `fixers/` for `RuleFix` builders, and `text/` for pure string helpers.
- There is no catch-all `utils/` folder. Flag any new `utils/` directory or any unrelated constants grouped in one file.
- Reuse an existing shared helper instead of re-implementing it. Flag duplicated logic; it should be extracted to the matching domain folder first.
- The shared folders are the bottom layer. Flag any import from `rules/` or `configs/` into a shared module.

## Rule files

- Each rule is built with `createRule` from `core/create-rule.js`. Flag any direct use of `ESLintUtils.RuleCreator`.
- One skeleton for every rule: external imports, local `.js` imports, `createRule({ name, meta, defaultOptions, create })`, then `export default`.
- `meta` must set `type`, `docs.description`, an explicit `schema` with `schema: []` when there are no options, and `messages` with camelCase `messageId` keys.
- The `name` passed to `createRule` must match the source filename, the registry key in `src/rules/index.ts`, the preset entry in `src/configs/`, and the doc filename in `docs/rules/`.
- A new rule must be wired in three places: the rule file, `src/rules/index.ts`, and the right preset in `src/configs/base.ts` or `react.ts`. It pairs with one test under `tests/rules/` and one doc under `docs/rules/`. A JSX or React rule must use `createRuleTester(true)`.

## Code quality

- Typed, small, focused units. No `any`, no dead code, no speculative abstraction.
- Tests mirror the source layout: a module under `src/<domain>/` is tested by `tests/<domain>/<name>.test.ts`.

## Docs and prose

- Parentheses are for code only — identifiers, commands, and signatures. Never put a requirement, caveat, example, or enumeration in parentheses; write it as a full sentence.
- Each rule doc under `docs/rules/` and the rules table in `README.md` are generated by `eslint-doc-generator`. Review the authored prose such as rationale and examples; do not flag the generated header, options table, or rules list.

## Artifacts stand alone

- Issues, pull requests, and docs must be self-contained. Never reference another repository, even as inspiration. Flag any cross-repository reference.

## Commits and pull requests

- Conventional-commit titles: `type(scope): subject`, type from the enum `build|chore|ci|docs|feat|fix|perf|refactor|revert|setup|style|test`, scope required, all lowercase, subject 50 characters or fewer, no body or footer.
- The PR title carries no `#N` reference; issue closures go in the body as `Closes #N`.
